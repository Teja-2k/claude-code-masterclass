Part 1 — Foundation

Chapter 01

What Claude Code Actually Is (and Isn't)

Claude Code is not another AI chatbot with a code editor bolted on. It is a terminal-native agentic coding tool built by Anthropic that runs directly in your shell, reads your files, executes commands, and writes code — all with your explicit permission. Understanding this distinction is the single most important thing that separates beginners from power users.

The CLI Architecture

When you launch Claude Code, you are starting an interactive agent session inside your terminal. Here is what actually happens under the hood:

  1. Your terminal becomes the interface. There is no browser, no Electron app, no VS Code panel. Your shell — the same one you use for git, npm, and ssh — is now your AI coding environment.
  2. Claude sees your filesystem. It can read files, list directories, search code, and understand your project structure. It has the same view of your codebase that you do.
  3. Claude executes real commands. When it needs to install a dependency, run a test, or start a dev server, it runs actual shell commands. Nothing is simulated.
  4. Everything requires permission. By default, Claude asks before reading sensitive files, writing changes, or executing commands. You are always in control.

How This Differs from ChatGPT, Copilot, and Cursor

Most AI coding tools follow a suggestion model: they predict what you might type next and offer autocomplete. This is useful for writing boilerplate, but it fundamentally limits what AI can do for you.

Claude Code follows an agent model: you describe what you want, and it figures out the steps, reads the relevant files, makes the changes, runs the tests, and iterates until the job is done. The difference is not incremental — it is categorical.

Feature Copilot/Cursor Claude Code
Interface IDE plugin Terminal CLI
Scope Current file Entire project
Can run commands Limited Full shell access
Multi-file edits Manual Automatic
Can test its own work No Yes

Why Terminal-First Matters

The terminal is the most powerful interface on your computer. Every deployment tool, every build system, every version control command, every server management utility — they all live in the terminal. By operating natively in this environment, Claude Code can:

  • Chain together complex workflows (git + build + test + deploy)
  • Work in any language, framework, or environment without plugins
  • Integrate with your existing toolchain seamlessly
  • Run on remote servers via SSH just as easily as on your local machine
  • Be scripted, automated, and composed into larger pipelines
Pro Tip

Claude Code works best when you think of it as a senior developer sitting at your terminal. You would not micromanage a senior developer line by line. You would describe the task, let them explore the codebase, and review their work when they are done.

Common Mistake

Treating Claude Code like a chatbot and asking it to "explain how to write a React component." Claude Code is a doing tool, not a teaching tool. Instead, say: "Create a React component that displays a user profile card with name, avatar, and bio."

Try This Now

Open your terminal and run claude. When the session starts, type: What files are in this directory? — watch how Claude Code actually reads your filesystem instead of guessing. Then ask it to create a simple file and observe how it asks for permission before writing.

Chapter 02

Installation & First 10 Minutes

Getting Claude Code running takes less than two minutes. Here is the exact process for every major platform.

Prerequisites

You need two things before installing Claude Code:

  • Node.js 18+ — Check with node --version. If you need to install it, use nodejs.org or your system package manager.
  • An Anthropic API key or a Claude Pro/Max subscription — Claude Code can authenticate through your Anthropic account directly or via an API key.

Installation

# Install Claude Code globally via npm
$ npm install -g @anthropic-ai/claude-code

# Verify the installation
$ claude --version
claude-code v1.x.x

# Start your first session
$ claude

On first launch, Claude Code will open your browser to authenticate. Sign in with your Anthropic account, grant permission, and you are ready to go. The authentication token is stored locally and you will not need to sign in again.

macOS-Specific Notes

# If you use Homebrew and don't have Node.js:
$ brew install node
$ npm install -g @anthropic-ai/claude-code

Windows-Specific Notes

Claude Code works on Windows through WSL2 (Windows Subsystem for Linux) or natively in PowerShell/Git Bash. WSL2 is strongly recommended for the best experience:

# In WSL2 (Ubuntu):
$ sudo apt update && sudo apt install nodejs npm
$ npm install -g @anthropic-ai/claude-code

# Or natively in PowerShell/Git Bash:
$ npm install -g @anthropic-ai/claude-code

Linux Notes

# Debian/Ubuntu:
$ curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
$ sudo apt install -y nodejs
$ npm install -g @anthropic-ai/claude-code

Your First 10 Minutes

Once installed, navigate to any project directory (or an empty one) and launch Claude Code:

$ mkdir my-first-project && cd my-first-project
$ claude

Here are the first five things to try, in order:

1. Ask Claude Code to explore

> What's in this directory? If it's empty, create a simple Node.js project with a package.json and an index.js that starts an Express server on port 3000.

2. Ask it to run the project

> Install the dependencies and start the server. Show me the output.

3. Ask it to modify code

> Add a /health endpoint that returns { status: "ok", timestamp: Date.now() }

4. Ask it to test

> Use curl to hit the /health endpoint and show me the response.

5. Ask it to commit

> Initialize a git repo and commit everything with a good commit message.

Understanding the Permission Model

Claude Code has a layered permission system designed to keep you in control:

  • Read permissions: Claude can freely read files in your project directory. It will ask before reading files outside the project root or sensitive files like .env.
  • Write permissions: Claude always asks before creating or modifying files. You will see a diff of what it wants to change and can approve or reject.
  • Execute permissions: Claude asks before running shell commands. You can approve individual commands or allow categories of commands (e.g., "allow all npm commands").

You can pre-configure permissions using a .claude/settings.json file in your project or your home directory:

// .claude/settings.json
{
  "permissions": {
    "allow": [
      "read",
      "bash(npm *)",
      "bash(node *)",
      "bash(git *)"
    ]
  }
}
Pro Tip

For projects you trust, create a .claude/settings.json that pre-allows common operations. This eliminates the "approve/deny" friction and lets Claude Code work at full speed. For unfamiliar codebases, keep the defaults — the approval prompts are your safety net.

Warning

Never blindly approve commands you don't understand. Claude Code will sometimes suggest running destructive commands like rm -rf or git reset --hard. Always read the command before approving. The permission system exists for a reason.

Try This Now

Create a fresh directory, launch Claude Code inside it, and walk through all five steps above. Pay attention to the permission prompts — notice how Claude asks before each write and execute operation. Try rejecting one request and giving different instructions to see how it adapts.

Chapter 03

The Mental Model

The number one reason people fail with Claude Code is that they use it like a chatbot. They type short, vague prompts and expect magic. The number one reason people succeed is that they learn to think in tasks.

Think in Tasks, Not Conversations

A conversation is open-ended. A task has a clear beginning, a set of concrete steps, and a verifiable end state. When you interact with Claude Code, you should always be assigning a task.

Bad prompt (conversation style):

> How do I add authentication to my app?

Good prompt (task style):

> Add JWT authentication to this Express app. Create a /auth/login endpoint that
  accepts email and password, validates against the users table in our SQLite
  database, and returns a JWT token. Add middleware that protects all /api/*
  routes. Use the jsonwebtoken package.

Notice the difference. The good prompt specifies:

  • What to build (JWT auth with login endpoint + middleware)
  • Where it goes (Express app, /auth/login, /api/* routes)
  • How it connects (SQLite database, users table)
  • What tools to use (jsonwebtoken package)

The Context Window Strategy

Claude Code operates within a context window — a limited amount of information it can hold in active memory at once. This is the most important technical constraint to understand, and managing it well is what separates intermediate users from advanced ones.

The key principles:

  1. Start fresh for new tasks. If you have finished adding authentication and now want to refactor the database layer, start a new session. Leftover context from the previous task will confuse the model and waste context space.
  2. Front-load critical information. Put the most important details at the beginning of your prompt. If there is a specific file Claude needs to look at, name it explicitly.
  3. Use the /clear command. Mid-session, if the conversation has gotten long and wandering, type /clear to reset the context without restarting. Then re-state your task clearly.
  4. Reference files instead of pasting code. Instead of pasting 200 lines of code into your prompt, say "Look at src/auth/middleware.js." Claude Code can read the file directly, and file reads are more token-efficient than pasted code.

The STAR Framework for Prompts

When you are not sure how to phrase a task, use the STAR framework:

  • Situation — What is the current state? ("I have an Express app with a SQLite database")
  • Task — What do you want done? ("Add JWT authentication")
  • Approach — Any specific method? ("Use the jsonwebtoken package, store tokens in httpOnly cookies")
  • Result — What does success look like? ("All /api routes should return 401 without a valid token")

Iterating Effectively

Even with perfect prompts, Claude Code will sometimes get things wrong. The key is efficient iteration:

# First attempt - be specific about the problem
> The login endpoint is returning 500. Check the error logs and fix the issue.

# Not this - too vague
> It's not working, fix it.

When iterating, describe the symptom, not the diagnosis. Say "the endpoint returns 500" rather than "I think there is a null pointer exception in the auth middleware." Let Claude Code investigate — it often finds root causes faster than you expect.

Pro Tip

The CLAUDE.md file is your secret weapon. Create a file called CLAUDE.md in your project root with notes about your project: the tech stack, coding conventions, important file locations, and anything Claude needs to know. Claude Code reads this file automatically at the start of every session.

# CLAUDE.md - Example

## Project: SaaS Dashboard
- Stack: Next.js 14, TypeScript, Prisma, PostgreSQL
- Style: Use Tailwind CSS, prefer server components
- Tests: Vitest for unit, Playwright for e2e
- Key dirs: src/app/ (pages), src/lib/ (utilities), prisma/ (schema)
- Convention: All API routes in src/app/api/, use zod for validation
- Deploy: Vercel, auto-deploy from main branch
Try This Now

Open a project you are working on (or create one). Write a CLAUDE.md file for it. Include your tech stack, key directories, coding conventions, and any gotchas. Then start a Claude Code session and give it a task — notice how it automatically picks up the context from CLAUDE.md and produces more aligned output.

Chapter 04

Your First Full Project

Theory is nothing without practice. In this chapter, we will build a complete web application from scratch using only Claude Code — from initializing the project to having a deployed, working app.

What We Are Building

A Bookmark Manager — a web app where users can save URLs with tags, search their bookmarks, and export them. It is simple enough to build in one session but complex enough to demonstrate real-world patterns.

Step 1: Project Initialization

$ mkdir bookmark-manager && cd bookmark-manager
$ claude

> Initialize a new Next.js 14 project with TypeScript and Tailwind CSS.
  Use the app router. Set up a SQLite database with Prisma. Create the
  following schema:
  - Bookmark: id, url, title, description, tags (string array), createdAt
  No authentication needed for now. Configure it so I can run "npm run dev"
  to start.

Claude Code will create the project structure, install dependencies, configure Prisma, create the schema, run the initial migration, and set everything up. Watch how it handles each step — creating package.json, running npx create-next-app, editing the Prisma schema, running npx prisma migrate dev.

Step 2: Build the Core UI

> Create the main page that shows all bookmarks in a card grid layout.
  Each card should show the title, URL (truncated), description preview,
  and tags as colored pills. Add a form at the top to add new bookmarks
  with fields for URL, title, description, and tags (comma-separated).
  Use server actions for the form submission. Make it look clean and modern
  with a dark theme.

Step 3: Add Search and Filtering

> Add a search bar that filters bookmarks by title, URL, or description
  in real time. Also add tag filtering - when you click a tag pill on any
  bookmark, it filters to show only bookmarks with that tag. Use URL
  search params so the filter state is shareable.

Step 4: Add Export Functionality

> Add an "Export" button that downloads all bookmarks (or filtered results)
  as a JSON file. Also add a "Export as HTML" option that generates a
  bookmark file compatible with browser import.

Step 5: Polish and Deploy

> Add loading states, error handling, and empty states to all components.
  Add a favicon and proper meta tags. Then set up the project for Vercel
  deployment - make sure it works with the SQLite database in production
  (or suggest a better approach for production).

At this point, Claude Code will likely suggest switching from SQLite to a hosted database for production (such as Turso, PlanetScale, or Vercel Postgres). This is exactly the kind of architectural insight that makes Claude Code valuable — it does not just do what you say, it flags problems.

The Full Build in Numbers

Building this project manually would take a competent developer 4-6 hours. With Claude Code, the entire build typically takes 20-30 minutes, including iteration and debugging. That is not a marginal improvement — it is an order of magnitude faster.

Pro Tip

After each major step, ask Claude Code to run the app and verify it works. Say: "Start the dev server, and then use curl to test the API endpoints" or "Start the dev server and take a screenshot." Catching issues early prevents cascading bugs.

Common Mistake

Trying to build the entire app in a single prompt. Even though Claude Code is powerful, breaking the work into logical steps (init, UI, features, polish, deploy) produces much better results. Each step gives Claude Code a clear, focused task with a verifiable outcome.

Try This Now

Follow the five steps above and build the Bookmark Manager. Do not copy-paste the prompts verbatim — adapt them to your preferred tech stack. If you prefer Vue, Svelte, or plain HTML/JS, tell Claude Code. The point is to experience the full cycle from zero to deployed app.

Chapter 05

Working with Existing Codebases

Building from scratch is fun. But most professional development happens inside existing codebases — often large, complex, and poorly documented. This is where Claude Code truly shines, because it can read and understand code faster than any human.

The Exploration Phase

When you point Claude Code at an existing project, start with exploration before making any changes:

> Analyze this project. Tell me:
  1. What framework and language it uses
  2. The overall architecture and directory structure
  3. How the data flows (database, API, frontend)
  4. Any obvious issues or technical debt you notice
  5. Where the main entry points are

Claude Code will read your package.json (or equivalent), scan the directory structure, sample key files, and give you a comprehensive overview. This alone can save hours of onboarding on an unfamiliar project.

Reading Strategies for Large Repos

For repositories with hundreds or thousands of files, Claude Code needs guidance on where to focus. Here are the most effective strategies:

Strategy 1: Top-Down Navigation

> Look at the project structure and identify the main entry point.
  Then trace the request flow for the /api/users endpoint from
  the route handler through the service layer to the database query.

Strategy 2: Search-Based Exploration

> Find all files that reference "stripe" or "payment" in this codebase.
  Then explain how the payment system works.

Strategy 3: Test-Guided Understanding

> Look at the test files to understand how the authentication system
  is supposed to work. Then look at the actual implementation and tell
  me if there are any gaps between the tests and the code.

Making Targeted Changes

The key to modifying existing code with Claude Code is specificity. You want to give Claude Code enough context to understand the change but narrow enough scope to avoid unintended side effects.

# Good: Specific file and function
> In src/services/userService.ts, modify the createUser function to
  also send a welcome email after creating the user. Use the existing
  emailService that's already imported.

# Bad: Vague scope
> Add email notifications to the app.

The "Understand Before Changing" Pattern

For complex changes, use a two-step approach:

# Step 1: Understand
> Read through the authentication middleware in src/middleware/auth.ts
  and all the files it imports. Explain how it works and what would need
  to change to add role-based access control.

# Step 2: Implement (in the same session, with context loaded)
> Go ahead and implement the role-based access control as you described.
  Add roles to the User model, update the middleware, and protect the
  admin routes in src/routes/admin.ts.

This pattern ensures Claude Code has thoroughly understood the existing code before it starts modifying it, dramatically reducing the chance of breaking changes.

Pro Tip

When working with large repos, create a CLAUDE.md that maps out the key directories and their purposes. Even a simple map like "src/models = database models, src/routes = API endpoints, src/services = business logic" dramatically improves Claude Code's ability to navigate the codebase.

Pro Tip

Use git diff integration after every change. Ask Claude Code: "Show me the git diff of all changes we've made." Review the diff before committing. This is your safety net against unintended modifications.

Try This Now

Clone an open-source project you have never worked with (try something like Formbricks or Cal.com). Launch Claude Code in the repo and ask it to analyze the architecture. Then ask it to make a small change — add a simple feature or fix a minor issue. Experience how Claude Code navigates a codebase it has never seen.

Chapter 06

The Debug Loop

Debugging is where most developers spend the majority of their time. It is also where Claude Code delivers the most dramatic time savings. The key technique is what I call the "describe the symptom" method.

The "Describe the Symptom" Technique

When you hit a bug, do not try to diagnose it yourself before involving Claude Code. Instead, describe exactly what you observe:

> When I click the "Save" button on the settings page, nothing happens.
  No error in the browser console, no network request is made. The button
  should submit the form and save the user's profile changes to the database.

This works better than "there is a bug in the form handler" because:

  • It does not bias Claude Code with a potentially wrong diagnosis
  • It gives Claude Code the expected behavior vs. actual behavior clearly
  • It lets Claude Code choose its own investigation strategy

The Debug Loop in Action

Claude Code follows a systematic debugging process:

  1. Reproduce: It looks at the relevant code to understand the expected flow
  2. Investigate: It traces through the code path, checking for obvious issues
  3. Hypothesize: It forms a theory about what is going wrong
  4. Test: It checks its hypothesis (reading more code, running commands)
  5. Fix: It implements the fix
  6. Verify: It runs tests or the app to confirm the fix works

Real Debugging Examples

Example 1: Server Error

> The /api/orders endpoint returns a 500 error when the cart has more
  than 10 items. It works fine with fewer items. Here's the error from
  the server logs: "PayloadTooLargeError: request entity too large"

Claude Code will immediately check the Express body parser configuration, find the default size limit, and increase it or implement pagination.

Example 2: UI Bug

> The modal component opens correctly but the close button doesn't work.
  Clicking outside the modal also doesn't close it. This used to work
  yesterday - the only change was adding the new toast notification system.

With the hint about the recent change, Claude Code will check for z-index conflicts, event propagation issues, or portal rendering problems caused by the toast system.

Example 3: Performance Issue

> The dashboard page takes 12 seconds to load. It used to load in under
  2 seconds. Run the database queries that the dashboard endpoint makes
  and tell me which one is slow. The endpoint is in
  src/app/api/dashboard/route.ts.

Using Error Logs Effectively

When you have error output, feed it directly to Claude Code:

> I get this error when running npm run build:

  TypeError: Cannot read properties of undefined (reading 'map')
    at ProductList (src/components/ProductList.tsx:24:31)
    at renderWithHooks (node_modules/react-dom/...)

  Fix this error.

Stack traces are gold for Claude Code. They point directly to the file, line number, and function where the error occurs, giving Claude Code an immediate starting point.

Pro Tip

If Claude Code fixes the immediate error but you suspect there might be related issues, ask: "Are there any other places in the codebase with the same pattern that could have the same bug?" Claude Code will search for similar code patterns and fix them proactively.

Common Mistake

Providing your diagnosis instead of the symptom. Saying "there is a null pointer in the auth middleware" leads Claude Code down a narrow path. Saying "authenticated users are getting redirected to the login page randomly" lets Claude Code cast a wider investigative net — the real cause might be a cookie expiration issue, not a null pointer at all.

Try This Now

Find a bug in one of your projects (or intentionally introduce one). Describe only the symptom to Claude Code and let it debug. Do not give hints about the root cause. Time how long it takes Claude Code to find and fix the issue compared to how long it would take you manually.

Chapter 07

Multi-File Orchestration

Real-world projects are not single files. A typical feature might touch the database schema, backend API, frontend components, tests, and documentation. Managing multi-file changes is where Claude Code's agentic nature pays off the most — and where most users leave performance on the table.

The Problem with Large Changes

When a change spans many files, the context window becomes a bottleneck. If Claude Code reads 20 files and then tries to edit them all, it may lose track of earlier files. The solution is structured decomposition.

Strategy 1: The Planning Prompt

Before making a large change, ask Claude Code to plan first:

> I want to add a "teams" feature to this SaaS app. Users should be
  able to create teams, invite members by email, and share resources
  within a team.

  Before writing any code, create a plan:
  1. What database schema changes are needed?
  2. What new API endpoints?
  3. What frontend components?
  4. What existing files need modification?

  Output the plan as a numbered list of steps.

Once you have the plan, execute it step by step:

> Execute step 1 from the plan: make the database schema changes.

Strategy 2: File-Scoped Instructions

Explicitly name the files you want changed:

> Make these changes across the codebase:
  - prisma/schema.prisma: Add a Team model and TeamMember join table
  - src/app/api/teams/route.ts: Create CRUD endpoints for teams
  - src/app/api/teams/invite/route.ts: Create invite endpoint
  - src/components/TeamSelector.tsx: Create team switcher component
  - src/lib/auth.ts: Add getCurrentTeam() helper

Strategy 3: Anchor Files

When working across many files, use an "anchor file" that ties everything together:

> Look at src/app/api/teams/route.ts as the starting point. Now make
  sure every file it imports from is implemented correctly, and that
  the frontend components that call these endpoints are updated to
  match the API contract.

Keeping Context Focused

Use the /clear command strategically between phases of a large change. A typical workflow:

  1. Session 1: Database schema + migrations
  2. /clear
  3. Session 2: Backend API endpoints
  4. /clear
  5. Session 3: Frontend components
  6. /clear
  7. Session 4: Tests and integration

Each phase gets a clean context with maximum space for the relevant files.

Pro Tip

After completing a multi-file change, always ask: "Run the linter and type checker across the entire project and fix any issues." This catches cross-file inconsistencies like mismatched imports, type errors, and missing exports that can slip through when changes span many files.

Try This Now

Take an existing project and plan a feature that touches at least 5 files. Use the Planning Prompt strategy to decompose it. Execute each step individually, using /clear between phases. Compare the result quality to what you would get trying to build it all in one prompt.

Chapter 08

Headless Mode & Automation

Everything we have covered so far involves interactive sessions — you and Claude Code, working together in real time. But Claude Code has a powerful headless mode that lets you run it non-interactively in scripts, CI/CD pipelines, and automated workflows. This is where Claude Code transforms from a coding assistant into a coding system.

The Basics of Headless Mode

You can pass a prompt directly to Claude Code from the command line:

# Single prompt, non-interactive
$ claude -p "Read package.json and list all dependencies that are outdated"

# Pipe input into Claude Code
$ cat error.log | claude -p "Analyze this error log and suggest fixes"

# Output in JSON format for programmatic use
$ claude -p "List all TODO comments in the codebase" --output-format json

Using Headless Mode in CI/CD

Here is a practical GitHub Actions workflow that uses Claude Code for automated code review:

# .github/workflows/claude-review.yml
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - run: npm install -g @anthropic-ai/claude-code

      - name: Run Claude Code Review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          DIFF=$(git diff origin/main...HEAD)
          claude -p "Review this code diff for bugs, security issues,
          and style problems. Be concise and actionable.

          $DIFF" --allowedTools "Read" > review.md

      - name: Post Review Comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review.md', 'utf8');
            github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: review
            });

Automation Scripts

Here are practical automation scripts you can use immediately:

Automated Dependency Updates

#!/bin/bash
# update-deps.sh - Run weekly via cron
cd /path/to/project
claude -p "Check for outdated npm dependencies. Update any that have
  minor or patch updates available. Run the test suite after updating.
  If tests pass, create a commit with the message 'chore: update
  dependencies'. If tests fail, revert and report which update broke
  the tests." --allowedTools "Bash,Read,Write"

Automated Changelog Generation

$ claude -p "Look at all commits since the last git tag. Generate a
  CHANGELOG entry grouped by: Features, Bug Fixes, Breaking Changes.
  Write it to CHANGELOG.md at the top of the file."

Code Quality Monitoring

$ claude -p "Analyze the codebase for: 1) Functions longer than 50 lines,
  2) Files with no test coverage, 3) Circular dependencies, 4) Unused
  exports. Output as JSON." --output-format json > quality-report.json

Security Considerations

When running Claude Code in automated pipelines, control its permissions carefully:

  • Use --allowedTools to restrict which tools Claude Code can use
  • Never give write or execute permissions in untrusted environments
  • Store API keys in CI/CD secrets, never in code
  • Review Claude Code's output before applying changes automatically
Warning

Be very careful with automated write permissions. A CI/CD pipeline that lets Claude Code modify files and push to your repo without human review is a security risk. Always require a human-in-the-loop for production changes.

Try This Now

Create a simple bash script that uses Claude Code in headless mode to analyze your project. Start with: claude -p "Give me a one-paragraph summary of this project and rate its code quality from 1-10". Then iterate to build a more sophisticated automated report.

Chapter 09

Custom Tools & MCP Servers

Out of the box, Claude Code can read files, write files, and run shell commands. But through the Model Context Protocol (MCP), you can extend it with custom tools that connect to databases, APIs, browser automation, and essentially anything with a programmatic interface.

What is MCP?

MCP (Model Context Protocol) is an open standard that lets AI models interact with external tools and data sources. Think of it like USB for AI — a universal connector that lets Claude Code plug into any system.

An MCP server is a small program that exposes tools (functions Claude Code can call), resources (data Claude Code can read), and prompts (pre-configured instructions). Claude Code can connect to multiple MCP servers simultaneously.

Configuring MCP Servers

MCP servers are configured in your Claude Code settings:

// .claude/settings.json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://user:pass@localhost:5432/mydb"
      }
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "ghp_your_token_here"
      }
    }
  }
}

Popular MCP Servers

  • Postgres/MySQL/SQLite — Query databases directly. Claude Code can run SELECT queries, inspect schemas, and help optimize queries.
  • GitHub — Create issues, review PRs, manage releases without leaving Claude Code.
  • Puppeteer/Playwright — Browser automation. Claude Code can take screenshots, fill forms, click buttons, and scrape data from web pages.
  • Filesystem (extended) — Access files outside the project directory with controlled permissions.
  • Fetch/HTTP — Make HTTP requests to external APIs.
  • Memory — Persistent knowledge graph that survives across sessions.

Building Your Own MCP Server

You can create custom MCP servers for your specific needs. Here is a minimal example using the TypeScript SDK:

// my-mcp-server/index.ts
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({
  name: "my-custom-tools",
  version: "1.0.0",
});

// Define a custom tool
server.tool(
  "check-uptime",
  "Check if a website is up and measure response time",
  { url: z.string().url() },
  async ({ url }) => {
    const start = Date.now();
    const res = await fetch(url);
    const duration = Date.now() - start;
    return {
      content: [{
        type: "text",
        text: `${url}: ${res.status} (${duration}ms)`
      }]
    };
  }
);

const transport = new StdioServerTransport();
await server.connect(transport);

Chapter 10

Building AI Agents with Claude Code

An AI agent is a system that can pursue goals autonomously — perceiving its environment, making decisions, taking actions, and iterating based on results. Claude Code is not just a tool you use to build agents; it is an agent. And you can use it to build other agent systems.

Agent Architecture Patterns

There are three primary patterns for building agents with Claude Code:

Pattern 1: The Loop Agent

An agent that runs in a loop, checking conditions and taking actions:

// agent-loop.ts - A monitoring agent
import { execSync } from "child_process";

function runClaudeCode(prompt: string): string {
  return execSync(
    `claude -p "${prompt}" --output-format json --allowedTools "Bash,Read"`,
    { encoding: "utf-8", timeout: 60000 }
  );
}

// Check every 5 minutes
setInterval(async () => {
  const analysis = runClaudeCode(
    `Check the application logs in /var/log/myapp/.
     If there are any ERROR level entries in the last 5 minutes,
     analyze the root cause and suggest a fix.
     Return JSON: { hasErrors: boolean, summary: string, suggestedFix: string }`
  );

  const result = JSON.parse(analysis);
  if (result.hasErrors) {
    // Send alert via Slack, email, etc.
    sendAlert(result);
  }
}, 5 * 60 * 1000);

Pattern 2: The Pipeline Agent

An agent that processes items through a series of stages:

// Each stage is a focused Claude Code call
const stages = [
  "Read the requirements document at docs/requirements.md",
  "Generate the database schema based on the requirements",
  "Create the API endpoints that the schema needs",
  "Generate test cases for each endpoint",
  "Run the tests and fix any failures",
];

for (const stage of stages) {
  await runClaudeCode(stage);
  // Each stage builds on the files created by previous stages
}

Pattern 3: The Multi-Agent System

Multiple Claude Code instances working on different parts of a problem:

// Run specialized agents in parallel
const [security, performance, accessibility] = await Promise.all([
  runClaudeCode("Audit this codebase for security vulnerabilities"),
  runClaudeCode("Profile the API endpoints and identify bottlenecks"),
  runClaudeCode("Check all pages for accessibility compliance"),
]);

// Combine results with a coordinator agent
const report = await runClaudeCode(
  `Combine these three audit reports into a single prioritized action plan:
   Security: ${security}
   Performance: ${performance}
   Accessibility: ${accessibility}`
);

The Subagent Pattern

Claude Code can also spawn subagents within an interactive session. When you are working on a complex task, Claude Code can delegate subtasks to focused agent instances:

> For each API endpoint in src/app/api/, create a comprehensive test
  file. Use a subagent for each endpoint so the work is parallelized.

Claude Code will dispatch focused tasks to subagent processes, each working in a narrow context, then collect and integrate the results.

Error Recovery

Robust agents handle failures gracefully. Always build retry logic and fallback strategies:

async function resilientAgent(task: string, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      const result = await runClaudeCode(task);
      return JSON.parse(result);
    } catch (error) {
      if (i === maxRetries - 1) throw error;
      // Refine the prompt on retry
      task = `Previous attempt failed with: ${error.message}. ${task}`;
    }
  }
}
Pro Tip

When building multi-agent systems, always have a "coordinator" agent that reviews the work of other agents. Without coordination, parallel agents can create conflicting changes. The coordinator merges results and resolves conflicts.
